System Overview

What is As-Shifa?

A security control demonstration system for a Healthcare Management platform. It implements 4 mitigation use cases to defend against real attacks on patient data.

Tech Stack

  • React 18 + TypeScript (frontend)
  • Vite (build tool)
  • Tailwind CSS + shadcn/ui (UI)
  • Role-Based Access Control (RBAC)
  • bcrypt password hashing simulation

Feature 1: Password Encryption

AUTH-05 MUC-01: Steal Login Credentials

What it does

  • Real-time password strength checker
  • 5 requirements: 8+ chars, uppercase, lowercase, number, special char
  • Password confirmation match validation
  • bcrypt hashing with salt on submit
  • Prevents plaintext storage & rainbow table attacks

Demo steps

  • Type weak password → see red X requirements
  • Type strong password (e.g. Admin@1234) → all green
  • Confirm password → match indicator
  • Click "Encrypt and Store" → bcrypt alert

Security Requirements Met


• AUTH-03: Password complexity enforced


• AUTH-05: Passwords stored as bcrypt hash


• CONF-01: Plaintext never stored


✓ 8+ characters
✓ Uppercase letter
✓ Lowercase letter
✓ Number
✓ Special character

Feature 2: Multi-Factor Authentication

AUTH-02 MUC-01: Steal Login Credentials

What it does

  • 3-step flow: Credentials → MFA code → Success
  • 3 MFA methods: Authenticator App, SMS, Email
  • 6-digit OTP input with auto-focus
  • Verification with loading state
  • Success screen confirms all security checks

Demo steps

  • Enter username: doctor@asshifa.com
  • Enter any password → click "Continue to MFA"
  • Select "Authenticator App" method
  • Type any 6 digits → "Verify and Login"
  • See success screen with auth checklist

Security Requirements Met


• AUTH-01: Credential verification


• AUTH-02: MFA for doctors & admins


• AUTHZ-01: Role-based access after login


• AUD-01: Login event logged


✓ Credentials verified
✓ MFA completed
✓ Role-based access granted
✓ Login event logged

Feature 3: Access Control (RBAC)

AUTHZ-01 to 04 MUC-02: Illegal Record Access

Role Rules

  • Patient: Own records only (AUTHZ-03)
  • Doctor: Assigned patients only (AUTHZ-02)
  • Administrator: Full access (AUTHZ-04)
  • Others: Denied — alert logged (AUD-05)

Demo steps (show 3 cases)

  • Dr. Sarah + R001 (Ahmad's record) → ✅ GRANTED
  • Dr. Mohammed + R001 (not his patient) → ❌ DENIED
  • Ahmad (patient) + R001 (own record) → ✅ GRANTED
  • Admin + any record → ✅ GRANTED

Security Requirements Met


• AUTHZ-01: RBAC enforced for all users


• AUTHZ-02: Doctors limited to assigned patients


• AUTHZ-03: Patients see own records only


• CONF-03: Unauthorized attempts blocked


• AUD-02, AUD-03: Access logged with timestamp

Feature 4: Audit Logs & Activity Monitoring

AUD-01 to 05 MUC-03 & 04: Record Tampering

What it does

  • Full activity log: 7 sample events
  • Search by user, action, resource ID
  • Filter by status: All / Success / Denied / Suspicious
  • Real-time suspicious activity banner
  • Click "View Details" → full event breakdown

Demo steps

  • Show full log — point to suspicious rows (amber)
  • Filter → "Suspicious Only" → 2 alerts
  • Click LOG-005 (brute force) → show IP, timestamp
  • Click LOG-007 (malicious insider) → show INT-04 alert
  • Search "Dr. Sarah" → filtered results

Security Requirements Met


• AUD-01: All activities logged


• AUD-02: User identity recorded


• AUD-03: Timestamps on all events


• AUD-04: Admin can review logs


• AUD-05: Suspicious activity alerts


• INT-02: Record modification history

Database Design

Users Table

PK user_id (VARCHAR)
name (VARCHAR NOT NULL)
email (VARCHAR UNIQUE)
role ENUM(patient, doctor, admin, insurance)
password_hash (VARCHAR) — bcrypt
created_at (TIMESTAMP)

Medical Records Table

PK record_id (VARCHAR)
FK patient_id → Users.user_id
FK assigned_doctor_id → Users.user_id
content (TEXT)
last_updated (TIMESTAMP)

Audit Logs Table

PK log_id (VARCHAR)
FK user_id → Users.user_id
action (VARCHAR) — AUD-01
resource_type (VARCHAR)
resource_id (VARCHAR)
result ENUM(success, denied, suspicious)
ip_address (VARCHAR) — AUD-02
AK timestamp (TIMESTAMP) — AUD-03

Primary Keys (PK): user_id, record_id, log_id

Foreign Keys (FK): patient_id, assigned_doctor_id, user_id

Alternate Keys (AK): email (unique), timestamp
