Threat Model & Mitigations

🔑
MUC-01 — Steal Login Credentials
Attacker brute-forces or steals doctor/admin passwords to access patient records
→
🔒 Password Encryption AUTH-05 🛡️ Multi-Factor Auth AUTH-02
📋
MUC-02 — Access Medical Records Illegally
Unauthorized user or wrong doctor attempts to read a patient's record
→
👥 Access Control RBAC AUTHZ-01–04
⚠️
MUC-03 & 04 — Modify Records / Fake Insurance Claim
Malicious insider tampers with medical records or submits fraudulent claims
→
📄 Audit Logs & Monitoring AUD-01–05
AUTH-05 MUC-01 Feature 1 of 4

Password Encryption

What it demonstrates

  • Real-time strength checker — 5 requirements
  • 8+ chars · uppercase · lowercase · number · special char
  • Password confirmation match validation
  • bcrypt hashing with salt on submit (AUTH-05)
  • Button disabled until all requirements pass

Demo script

1Type abc → see red ✗ for 4 requirements
2Type Admin@1234 → all turn green
3Confirm field → match indicator appears
4Click "Encrypt and Store" → bcrypt confirmation

Requirements implemented

AUTH-03Password complexity policy enforced before storage
AUTH-05Passwords stored as bcrypt hash with per-user salt
CONF-01Plaintext password never stored or transmitted

Test data

Valid: Admin@1234
Invalid: abc, password, 12345678

AUTH-02 MUC-01 Feature 2 of 4

Multi-Factor Authentication

What it demonstrates

  • 3-step flow: Credentials → OTP code → Success
  • 3 methods: Authenticator App, SMS, Email
  • 6-digit OTP input with auto-focus between cells
  • Wrong code → error shown (rejects invalid input)
  • Correct code 123456 → success with auth checklist

Demo script

1Enter doctor@asshifa.com + any password
2Select "Authenticator App" method
3Type 000000 → see "Invalid code" error
4Type 123456 → see success screen

Requirements implemented

AUTH-01Credential verification before second factor
AUTH-02MFA required for doctors and administrators
AUTHZ-01Role-based access granted only after MFA
AUD-01Login event logged on success

Test data

Correct OTP: 123456
Wrong OTP: 000000, 999999, any other 6 digits

AUTHZ-01–04 MUC-02 Feature 3 of 4

Access Control (RBAC)

Role rules

  • Patient — own records only (AUTHZ-03)
  • Doctor — assigned patients only (AUTHZ-02)
  • Administrator — full access (AUTHZ-04)
  • Others — denied, alert logged (AUD-05)

Demo script — show 4 cases

1Dr. Sarah + R001 → GRANTED (assigned)
2Dr. Mohammed + R001 → DENIED (not his patient)
3Ahmad (patient) + R001 → GRANTED (own record)
4Admin + any record → GRANTED (full access)

Requirements implemented

AUTHZ-01RBAC enforced for every access request
AUTHZ-02Doctors limited to their assigned patients
AUTHZ-03Patients can view only their own records
AUTHZ-04Administrators have unrestricted access
CONF-03Unauthorized attempts blocked and logged

  • Live search by user name, action, resource ID
  • Filter by status: All / Success / Denied / Suspicious

Database Design

Users
PK user_id VARCHAR
name VARCHAR NOT NULL
email VARCHAR AK
role ENUM(patient, doctor, admin, insurance)
password_hash VARCHAR
created_at TIMESTAMP

password_hash stores bcrypt output — plaintext never saved (AUTH-05)

Medical Records
PK record_id VARCHAR
FK patient_id → Users
FK assigned_doctor_id → Users
content TEXT
last_updated TIMESTAMP
version INT DEFAULT 1

assigned_doctor_id enables AUTHZ-02 — doctors only see their patients

Audit Logs
PK log_id VARCHAR
FK user_id → Users
action VARCHAR — AUD-01
resource_type VARCHAR
resource_id VARCHAR
result ENUM(success, denied, suspicious)
ip_address VARCHAR — AUD-02
timestamp TIMESTAMP AK

KEY LEGEND

PK Primary key   FK Foreign key   AK Alternate key
